Code review structure:
    1) Context:
        - purpose. 
        - inputs/outputs.
        - class responsibilities. 
        - where/what are the complex bits. 

    2) Design - issue + suggest solution:
        - god classes.
        - Coupling. 
        - seperation of concerns.
        - abstraction. 
        - lack of enums/interfaces

    3) Readbility issue + suggest solution:
        - vague + inconsistant naming
        - long methods
        - deep layers/nesting
        - magic numbers
        - bad commenting
        - poor formatting

    4) Robustness issue + suggest solution:
        - Null handling
        - error handling clarity
        - missing validations
        - ignring exceptions
        - threading/concurrency problems

    5) Maintainability + testability issue + suggest solution:
        - Can we test easily?
        - duplications?
        - lack of modularity
        - too many side-effect actions
        - state required.
        - hard to change/extend  

        FORMAT:

    ISSUE        WHY             FIX                 REGRESSION TEST
1)  god class    hard to         Extract to a, b,    tests for each new class. 
                 maintain.       c etc.
    

After CL is made and implemented peer review the changes by asking:
        - Can I still understand the order flow?
        - Did the main method get smaller?
        - Does each class have a clear reason to exist?
        - Are business rules easier to find?
        - Do tests still pass after the change?
        - Was a design pattern used because it helped, or because it sounded good?
